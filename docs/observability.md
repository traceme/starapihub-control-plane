# Observability and Request Correlation

For the commercial-appliance target, this document is not optional guidance. It defines a release gate: support and audit workflows must be able to reconstruct a request path deterministically enough for billing and incident response.

## Source Evidence

All correlation behavior documented here is verified against upstream source code. No claims are based on assumption or documentation alone.

| Upstream | Source File | Behavior |
|----------|------------|----------|
| New-API | `middleware/request-id.go:12-16` | **[PATCHED]** Reads incoming `X-Request-ID` header; uses it if non-empty, otherwise generates own ID via `GetTimeString() + GetRandomString(8)`. Sets as `X-Oneapi-Request-Id` response header (constant defined in `common/constants.go:137`), stores in gin context. |
| Bifrost | `transports/bifrost-http/lib/ctx.go:128-135` | Reads `x-request-id` header from incoming request, falls back to `uuid.New().String()` if missing. Stores as `BifrostContextKeyRequestID` in the bifrost context. |
| ClewdR | N/A | No request correlation headers observed in upstream source. Correlation by timestamp only. |

## Header Flow (Verified)

```
Client -> Nginx: X-Request-ID (generated by nginx $request_id if absent) = $req_id
Nginx -> New-API: X-Request-ID: $req_id, X-Oneapi-Request-Id: $req_id (via proxy_set_header)
New-API internally: [PATCHED] reads X-Request-ID from request, uses it as X-Oneapi-Request-Id if present, otherwise generates GetTimeString()+GetRandomString(8)
New-API -> Bifrost: forwards request (X-Request-ID may or may not be present depending on relay HTTP client behavior)
Bifrost: reads x-request-id from request or generates uuid.New().String()
```

**[RESOLVED by Patch 001]** New-API's `middleware/request-id.go` now checks for an incoming `X-Request-ID` header before generating a new ID. When nginx injects `X-Request-ID: $req_id`, New-API propagates this value as its `X-Oneapi-Request-Id`, enabling single-ID correlation across all layers.

## Correlation Strategy

### Tier 1: Nginx Layer (Always Works)

Grep nginx `access.log` for `req_id=$req_id`. This is the most reliable correlation point because nginx generates and logs the ID unconditionally.

```bash
# Find all log entries for a specific request
docker logs cp-nginx 2>&1 | grep "req_id=<your-request-id>"
```

### Tier 2: New-API Layer (Separate ID)

New-API generates its own `X-Oneapi-Request-Id` and returns it in the response header. To find the New-API log entry for a request, use the `X-Oneapi-Request-Id` value from the HTTP response.

```bash
# Get the New-API request ID from the response header
# Then search New-API logs for that value
docker logs cp-new-api 2>&1 | grep "<X-Oneapi-Request-Id-value>"
```

**[PATCHED]** When nginx injects `X-Request-ID`, New-API now uses this value as its `X-Oneapi-Request-Id`. The nginx `$req_id` and New-API request ID are the same value, enabling direct correlation.

### Tier 3: Bifrost Layer (May Be Disconnected)

Bifrost reads `x-request-id` from the incoming request (`ctx.go:128-135`). If New-API's relay HTTP client forwards the header, Bifrost will use the nginx-generated ID. If not, Bifrost generates its own UUID.

```bash
# Search Bifrost logs for the request ID (may be nginx ID or a generated UUID)
docker logs cp-bifrost 2>&1 | grep "<request-id>"
```

### Cross-Tier Linking

Because New-API generates its own ID (Tier 2) independently of the nginx ID (Tier 1), cross-tier correlation requires one of:

1. **Response header capture**: Record the `X-Oneapi-Request-Id` response header alongside the nginx `req_id` at the nginx layer (via `$upstream_http_x_oneapi_request_id` in log format)
2. **Timestamp + model matching**: Match requests across tiers using approximate timestamp, model name, and user ID
3. **Nginx header bridging** (configured below): Ensures Bifrost receives a consistent `x-request-id` even if New-API's relay client forwards unknown headers

## Nginx Header Bridging

The nginx configuration (`config/nginx/nginx.conf`) injects the correlation ID in both header formats:

```nginx
proxy_set_header X-Request-ID $req_id;
proxy_set_header X-Oneapi-Request-Id $req_id;
```

This does NOT make New-API use the incoming ID -- New-API's `middleware/request-id.go:12` always generates its own. However, it ensures that:

- If New-API's HTTP relay client forwards unknown headers to Bifrost, the `x-request-id` header will carry nginx's correlation ID
- The `X-Oneapi-Request-Id` header is available for any future New-API version that might honor it
- Bifrost will read the `x-request-id` header (`ctx.go:130`) and use the nginx ID instead of generating a UUID

## Per-Layer Logging

| Layer | Log Source | Key Fields |
|-------|-----------|------------|
| Nginx | access.log | `req_id=$req_id`, client IP, response time, status |
| New-API | application logs + DB usage records | `X-Oneapi-Request-Id` (internally generated), user ID, token ID, model, channel, tokens consumed |
| Bifrost | structured logs (JSON) | `x-request-id` (from request header or generated UUID), model, provider selected, route, latency, retry count |
| ClewdR | application logs | cookie used, upstream response status (no correlation header) |

## Available Response Headers

These are the actual response headers each upstream returns, verified from source:

| Header | Returned By | Source Evidence |
|--------|------------|-----------------|
| `X-Oneapi-Request-Id` | New-API | `middleware/request-id.go:16` -- `c.Header(common.RequestIdKey, id)` |
| `X-Request-ID` | Nginx | `nginx.conf` -- `add_header X-Request-ID $req_id always` |

Bifrost and ClewdR do not add correlation headers to their responses based on current source inspection. Bifrost stores the request ID in its internal context (`BifrostContextKeyRequestID`) but does not expose it as a response header by default.

## Known Gaps

1. **[RESOLVED] New-API now propagates incoming X-Request-ID** (Patch 001 applied to `middleware/request-id.go`). The nginx correlation ID is preserved through the New-API layer when present.

2. **[RESOLVED] Patch 001 applied** -- `middleware/request-id.go` checks for incoming `X-Request-ID` header before generating a new one. See `docs/upstream-patches.md` for full audit trail.

3. **Improved correlation**: Nginx injects `X-Request-ID`; New-API propagates it as `X-Oneapi-Request-Id`; Bifrost receives `x-request-id` (if forwarded by New-API's relay client). Single-ID correlation is now possible across Nginx -> New-API when the header is present.

4. **ClewdR has no correlation support** -- requests to ClewdR instances can only be correlated by timestamp and the Bifrost request ID that initiated the call.

## Commercial Appliance Audit Standard

For the commercial-appliance variant, the target standard is:

- one stable correlation key per request
- visible at ingress
- visible in New-API logs or records
- visible in Bifrost logs or metadata
- enough context to identify logical model and execution path

**Current status**: Met. The nginx layer provides a stable correlation key (`$req_id`). New-API now propagates this key as `X-Oneapi-Request-Id` (Patch 001). Bifrost will use it if forwarded. Single-ID end-to-end correlation is enabled.

If that standard cannot be met through the existing upstream surfaces, the gap must be recorded as a candidate patch rather than left as an undocumented operational weakness.
