# CEO Plan: StarAPIHub Command Center Dashboard

## Vision

### 10x Check

The baseline Phase 2 is "a dashboard for policy CRUD and config sync." That's a settings page. The 10x version is the **StarAPIHub Command Center** — a near-real-time operations dashboard that makes managing an AI API gateway feel like piloting a spaceship.

The operator opens one tab and sees: system health, live traffic, cookie pool status, model usage, recent alerts. They never touch a YAML file. They never grep docker logs. They never SSH into the server to check if things are working. The dashboard is the product.

### Platonic Ideal

- **Zero YAML editing.** Every config change happens in the UI with live preview and validation.
- **Know before your users.** Cookie expiring? Alert. Provider outage? Auto-failover + notification. Rate limit approaching? Preemptive rotation.
- **Dashboard feels alive.** Near-real-time traffic view (5-second polling). See request rates, latencies, and provider health update live.
- **Zero-config onboarding.** Setup wizard: add key → create model → test request → done in 2 minutes.
- **API-first design.** The dashboard is a client of a well-designed REST API. CLI parity is a future phase; Phase 2 delivers the API that would power it.

## Accepted Scope (Phase 2 = Command Center)

### Core Service
- **Go HTTP service** (`control-plane-api`) with REST API
- **SQLite** for local state (alert config, wizard completion, cached metrics) — WAL mode enabled, single-writer connection pool via `modernc.org/sqlite` pure-Go driver
- Integrated into docker-compose as `cp-dashboard` on **core-net + provider-net** (needs direct access to ClewdR for cookie polling)
- **React + Vite** frontend served as static files by the Go process (same-origin, no CORS needed)
- Frontend refreshes via **SSE (Server-Sent Events)** for health/traffic updates (5-second push interval)

### Authentication
- **Static admin token** from environment variable (`DASHBOARD_TOKEN`) for single-operator use
- Token required for all `/api/*` endpoints (Bearer auth)
- Future phase: integrate with New-API's user system for multi-user access control

### Polling & Staleness
- Health polls: every **10 seconds** to New-API `/api/status`, Bifrost `/health`, ClewdR `/api/cookies`
- Cookie polls: every **60 seconds** (ClewdR `/api/cookies?refresh=true` bypasses 5-min cache)
- Log aggregation: tails nginx access log file (mounted as volume) every **5 seconds**
- **Stale data display:** if a service hasn't responded in >30 seconds, show last-known value with amber "stale" badge and timestamp

### Alerting
- Phase 2 alert channels: **in-dashboard banner** + **optional webhook URL** (configurable via env var `ALERT_WEBHOOK_URL`)
- Webhook payload: JSON with alert type, service, message, timestamp
- Future phase: email, Slack, Telegram integrations

### Feature Set
1. **Near-real-time health dashboard** — request rate, p50/p99 latency, error rate, per-provider breakdown. Data from nginx access log (tailed) and upstream health endpoints.
2. **Cookie health panel** — per-instance valid/exhausted/invalid, utilization meters, 80% threshold alerts, one-click link to ClewdR admin UI for rotation
3. **Visual model routing editor** — CRUD for logical models. "Drag providers to reorder priority within a route's fallback chain." Live preview of routing result. Compare current vs. pending config before applying.
4. **Setup wizard** (conditional on design partner feedback) — 4-step onboarding: add API key → create model → test request → show API token
5. **Request log viewer** — recent requests with status/latency/model, click-to-trace with X-Request-ID correlation. **Limitation:** correlation is only as deep as upstream services propagate X-Request-ID. Nginx → New-API propagation is confirmed. Bifrost and ClewdR propagation must be verified at implementation time; if not supported, trace shows nginx-level data only.

### Config Sync Mechanisms
- **Bifrost:** call Bifrost admin API (`/api/providers`, `/api/config`, `/api/governance/virtual-keys`) — these endpoints were discovered and verified during Phase 1
- **New-API:** call New-API admin API (channel CRUD, model/option endpoints) — also verified during Phase 1
- **No file-based sync needed** — both services have functional HTTP APIs for all config operations

### Architecture

```
                    ┌──────────────────────────────┐
                    │   cp-dashboard :8090          │
                    │   (Go + React static files)   │
                    │                               │
                    │  /api/health    → polls upstreams (10s)
                    │  /api/cookies   → polls ClewdR (60s)
                    │  /api/models    → CRUD → syncs via API
                    │  /api/wizard    → orchestrates setup
                    │  /api/logs      → tails nginx access log
                    │  /api/sse       → SSE stream for frontend
                    │  /              → serves React frontend
                    └──────┬──────────┬────────────┘
                           │          │
                      core-net    provider-net
                           │          │
              ┌────────────┼──────┐   │
              │            │      │   │
        [new-api]    [bifrost]  [pg] [redis]
                                      │
                               [clewdr-1/2/3]
```

### Control Plane Failure Semantics (unchanged)
Dashboard down = data path continues with last-known config. Dashboard is never on the hot path.
